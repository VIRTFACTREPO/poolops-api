PoolOps
Master UI Specification

Complete build reference for all screens across Mobile (Technician), Mobile (Pool Owner) and Web Dashboard (Admin). This document is the authoritative source for all UI decisions, component behaviour, navigation flows and interaction patterns. Charlie builds directly from this spec.
Version 1.0  ·  April 2026  ·  MVP Scope


## [Heading1] 1. Platform Overview
PoolOps is a purpose-built operations platform for pool maintenance businesses. It replaces paper dockets, WhatsApp scheduling and spreadsheet records with a single platform that understands pool chemistry, service workflows and compliance requirements.


## [Heading2] 1.1 Architecture
Two surfaces, three user types.

Surfaces
Mobile app
iOS + Android. Role determined at login. Technician sees run sheet and service tools. Pool owner sees service history and booking options. Same app, different views.
Web dashboard
React / Next.js. Dark mode by default. Full operational portal for business owner, operations manager and admin staff. One login, all features.

User types
Pool Technician
Field worker. 10–20 pools per day. Uses mobile exclusively. Run sheet, chemical logging, job completion, photo capture.
Pool Owner
Customer. Uses mobile to view service history, reports, and request extra visits.
PoolOps User
Business owner / ops / admin. Uses web dashboard for scheduling, customers, records, team and settings.

## [Heading1] 2. Design Language

## [Heading2] 2.1 Aesthetic direction
Refined operational minimalism. The product should feel like a precision instrument — calm, uncluttered and immediate. Every element earns its place. The reference aesthetic is the Velix AI Agent dashboard: near-white neomorphic card language, muted accent states, near-black active indicators, and icon-only navigation with a dark pill active state.
The mobile app uses the light mode palette throughout. The web dashboard defaults to dark mode for reduced eye strain during long operational use.

## [Heading2] 2.6 Navigation patterns

## [Heading3] Mobile — bottom nav
Icon-only. Active item: dark pill background (#111827) with white icon. Inactive: #9CA3AF icon, no background. Home pill (iOS indicator) always below nav on white background.
Bottom nav rules
Visible on
Top-level screens only: M4 Run Sheet, M13 Notifications, M14 Profile (technician); M15 Home, M16 History, M21 Notifications (owner)
Hidden on
All drill-down screens: M5, M6 and all tabs, M17, M18. These use a back arrow in the header instead.
CTA conflict
Any screen with a sticky primary CTA at the bottom does not show the nav bar. The CTA owns that space.

## [Heading1] 3. Technician Mobile Screens
The technician app is the primary field tool. All screens prioritise speed and one-handed operation. Every primary action is a large touch target. No screen requires more than 3 taps to complete its core task.

## [Heading2] M2 — Login
Full screen, vertically centred, #F5F5F3 background.
Layout
Logo block
Droplet icon mark (32px, #F5F5F3 on #111827 rounded square) + "PoolOps" wordmark + tagline "Pool smarter. Always." in #9CA3AF. Centred, occupies upper 40% of screen.
Form block
Email input, password input (with eye toggle), LOG IN primary button, Forgot password link in #0EA5E9.
Footer note
Divider + "New pool owner? Wait for invite from your pool company." in #9CA3AF. Prevents confusion about self-registration.
Behaviour
Role routing
No role selector on screen. Role is set by admin when creating account. On successful login, server returns role. Technician routes to M4, pool owner routes to M15.
Error state
Both fields get red border. Error message below button: "Incorrect email or password" in #EF4444.

## [Heading2] M10 — Job Notes (Tab)

## [Heading1] 7. Implementation Rules

## [Heading2] 7.1 Buttons and CTAs
Never use CSS class inheritance for button text colour or background. Always set inline.
Use div elements for sticky CTAs, not button elements, to avoid browser default style conflicts.
All sticky footers must have explicit background-color: #ffffff (light) or #0F172A (dark). Never transparent or inherited.
Primary CTA icon SVG stroke must be set to #ffffff explicitly on the SVG attribute.
Test every button in the browser without hover to verify colour is correct at rest.


## [Heading2] 7.2 Mobile screens
All phone frames use border-radius: 50px with overflow: hidden.
Dynamic island: 120px wide, 34px tall, #111827, border-radius: 20px. Always centred.
Home pill: 120px wide, 4px tall, #111827. Always on white background below nav.
Status bar icons always rendered as inline SVG, never as images.
Bottom nav height: 10px top padding + icon area (32px) + 14px bottom padding. White background.
Safe area: home pill area provides bottom safe area buffer. Do not pad separately.


## [Heading2] 7.3 Colour rules
Never use Pool Blue (#0EA5E9) as a primary action colour on mobile. Use #111827.
Pool Blue is used only for: links, active tab underlines, selected states, connection status pills.
Traffic light indicators always paired with text. Never colour alone.
Amber access notes card: background #FFFBEB, left border 4px #F59E0B, border #FDE68A.
Customer note / office note are visually distinct. Customer = blue-tinted border. Office = red-tinted.


## [Heading2] 7.4 Web dashboard
Sidebar is position: fixed, not sticky. Main content has margin-left: 220px.
Topbar is position: sticky, top: 0, z-index: 10.
Dark mode translucent colour fills for pills (rgba not solid hex) to sit naturally on dark surfaces.
Table rows: hover background #243447. Cursor pointer on clickable rows.
Breadcrumb in topbar on all drill-down screens.
Forms max-width: 720px (add/edit customer), 520px (add/edit technician). Left-aligned, not centred.


## [Heading2] 7.5 Navigation rules
Bottom nav hidden on all drill-down screens. Back arrow in header handles return.
Sticky CTA and bottom nav never coexist. CTA owns the bottom of drill-down screens.
Active job (M6) back button shows abandon confirmation, never immediately navigates.
Web breadcrumbs: parent is clickable and muted (#64748B). Separator is ›. Current page is white.


## [Heading2] 7.6 Data and state
Pool owner never sees office notes. This must be enforced at the API level, not just UI.
Service records are locked after submission. Web users can only add/edit office notes post-lock.
Stock deduction happens on Mark Complete, not when treatment amounts are entered.
Flagged readings trigger auto-alerts on job completion. Inbox badge increments, dashboard updates.
Schedule changes made by admin auto-push to technician run sheet with toast notification.
